## 3. 핵심 의사결정 타임라인

### 3-1. 기획 단계
- **주제 확정** — "AI 기반 양액 펌프 막힘 및 이상 감지" (스마트팜 3대 페인 포인트 중 데이터 기반 자율 제어 영역 — [DOMAIN_KNOWLEDGE.md §6](DOMAIN_KNOWLEDGE.md))
- **데이터 출처 결정**:
  - 메인 — AI-hub 딸기 재배환경 센서 데이터 (기초 통계 추출)
  - 보조 — Water Pump Sensor Data (Kaggle, 비교/프로토타입용. 실제 학습 미사용)
- **핵심 가설** — 화학 센서(EC/pH)는 데이터 품질 이슈 잦음 → 모델 입력에 포함할지 여부를 비교 실험 후 결정
- **핵심 가설 2** — 물리 센서(유량/압력/모터부하/RPM) 기반 다변량 이상탐지가 단일 임계치보다 우월

### 3-2. 데이터 합성 단계
- AI-hub 데이터엔 **펌프·유압 데이터가 없음** → 자료조사 기반 가상 데이터 생성기 자체 구현
- **45개 변수 × 1분 단위 × 3개월 분량** raw data 합성 (clog-focused 데이터셋)
- 별도 테스트 데이터 생성기(`data_gen_test.py`)로 `anomaly_label` 포함 평가셋 구축
- **EC/pH 학습 배제 결정** — 화학 센서 신뢰도 이슈로 학습 입력에서 제외 (가상 데이터엔 포함되어 있음)

### 3-3. 모델링 단계 핵심 결정
1. **4개 도메인 독립 AE** — 단일 모델 대비 도메인 간 오류 전파 차단 (motor / hydraulic / nutrient / zone_drip)
2. **SHAP 앙상블 자동화** — 타겟별 K-means 배경 압축 + RF 샘플링 + SHAP 계산 → 교집합/voting/union 3단계 피처 셋 자동 도출
3. **VIP 피처 주입 레이어 분리** — `feature_engineering.py`에서 Scaler 앞단에만 컨텍스트 피처 주입 (Tier 3 LSTM-AE 전환 시에도 재사용 가능)
4. **알람 근거·설명 정합화** — 시간/운전상태 컨텍스트 피처 12개를 MSE 점수와 RCA에서 일괄 제외, train과 inference가 동일 피처 셋 공유 (단일 소스 `DEFAULT_CONTEXT_FEATURES`)
5. **기동 스파이크 알람 억제** — `is_startup_phase=1` 구간은 alarm_level=0 강제
6. **6-Sigma threshold** — mean + k·std 기반 Caution(2σ)/Warning(3σ)/Critical(6σ) 3단계 자동 산출
7. **정량 평가 루틴 구현** — `evaluate_test_metrics.py`로 anomaly_label 기반 precision/recall/F1/FAR + FN/FP 타임스탬프 CSV 저장

### 3-4. 서비스 통합 단계
- FastAPI 기반 실시간 추론 API `/predict` — 4개 도메인 동시 추론 + 통합 알람 레벨 산출
- 추론 응답에 per-feature 3-시그마 밴드·실제값·기대값 포함 (프론트 차트 직결)
- Docker Compose 기반 배포 환경 구성 (inference-api, frontend 분리)
- 클라이언트 시뮬레이터로 end-to-end 재현 가능

## 4. 실패에서 얻은 교훈

### 4-1. Phase A-1 — 희소 binary 입력 실패
**시도** — `is_startup_phase` (기동 직후 5분 플래그)를 AE 입력에 포함
**결과** — **MSE 50배 폭발**. 0.7%만 `=1`인 희소 binary가 학습을 망침
**교훈** — **"희소 binary 금지" 규칙 수립**. 운전 맥락은 연속값(`minutes_since_startup`)으로 전달
**적용 위치** — [feature_engineering.py:24-36](../services/inference/src/feature_engineering.py#L24-L36) 주석에 재활성화 조건 명시 (sample_weight 가중 또는 oversampling으로 비율 ≥ 10%)

### 4-2. Phase A-3 — nutrient raw 센서 target 재편 시도
**시도** — nutrient 도메인 target을 raw 센서 기반으로 재편
**결과** — 재학습 후 **전 도메인 악화** (motor F1 0.527 → 0.106, zone_drip 완전 붕괴)
**교훈** — **A-2 상태로 롤백**. 원인 미규명 → 추후 재검토 대기
**적용 위치** — [train.py:260-262](../services/inference/src/train.py#L260-L262) 주석으로 사례 기록

### 4-3. flow_drop_rate 분모 발산 버그
**시도** — `(A - B) / (B + ε)` 꼴 공식, ε는 0-division 방지용
**결과** — 학습 데이터 0.5% 샘플이 **수천만 단위 극단값** → MinMaxScaler 왜곡 → AE MSE 폭발
**교훈** — **"ε는 0-division 방지용일 뿐, 수치 안정화가 아니다"**. 분모 발산 방지 게이트 의무화
**현재 적용** — 3단 게이트(`pump_on`, `min baseline`, `[0,1] clipping`)로 차단

### 4-4. 알람 해석성 — 컨텍스트 피처 오염
**문제** — RCA Top이 `time_cos`, `time_sin` 같은 **액션 불가능한 피처로 채워짐**
**해결** — 컨텍스트 피처 12개를 점수·설명에서 일괄 제외 (단일 소스 `DEFAULT_CONTEXT_FEATURES`)
**원칙 수립** — **"알람 근거 = 설명"** — 점수와 RCA가 동일 피처 셋 공유

## 5. 진행 중 문제 (2026-04-23 기준)

### 5-1. NUTRIENT 도메인 오탐 독점 ⚠️
**현황** — EVAL 구간 FP **581건 중 547건(94%)이 nutrient**. **단독으로 잡은 진짜 이상은 0건**
**임시 운영** — overall 알람 계산에서 nutrient 제외 ([evaluate_test_metrics.py:41](../services/inference/src/evaluate_test_metrics.py#L41) `EXCLUDE_FROM_OVERALL`)
**근본 수정 후보** — 다음 3안 중 실험 선택 예정:
- (a) SHAP voting `top_ratio` 상향 (0.25 → 0.4)
- (b) `df_interpret`에 양액 원천 센서 passthrough + `NUTRIENT_VIP` 강제 주입
- (c) robust voting 대신 도메인별 명시 피처 리스트 전환

### 5-2. train.py 비결정성
**현상** — `random_state=42` 설정에도 재학습마다 다른 모델 산출
**원인 후보** — RF `n_jobs=-1`, SHAP 샘플링, KMeans 랜덤성 의심 (미규명)
**대응 계획** — 전역 seed 고정, RF `n_jobs=1` 실험, SHAP/KMeans 랜덤성 감사

### 5-3. 모델 아티팩트 소실 리스크
**현황** — `/models`가 `.gitignore`되어 있고 재학습이 덮어씀
**대응 계획** — 재학습 전 스냅샷 규칙 수립 (예: `models_phaseD_2026-04-23/` 형태로 백업)

### 5-4. 알람 사양 미적용
**현재 운영** — 시그마 레벨 (2, 3, 6)
**목표 사양** — **(1, 2, 3) σ + 도메인별 연속 N회 디바운싱** (Caution 3회 / Warning 5회 / Critical 7회 초안)
**이유** — 강사님 피드백 반영 + 스파이크성 노이즈 차단
**상태** — 미구현 (디바운싱 로직 추가 필요)

### 5-5. 프론트–백엔드 완전 연동 지연
**현황** — MVP 범위 조정 필요
**대응 계획** — 대시보드에 "도메인별 상태 지표 / RCA Top3 / 알람 이력 / 스파이크 태그" 표시 흐름 확정 + 기동 스파이크(🔄) vs 이상 스파이크(⚡) 구분 UI

## 7. 강사님 피드백 정리

### 7-1. 받은 피드백 요지
- **As-Is / To-Be 형식**으로 문제 현상·원인·조치·공수비용 정리
- **펌프 시스템 매커니즘** 설명 추가 ([DOMAIN_KNOWLEDGE.md §1-3](DOMAIN_KNOWLEDGE.md))
- **막힘 현상의 종류와 원인**, **월간 발생 빈도, 비용/인력 손실** 정량화
- **MSE 정상 분포 기반** 임계치 설명 강화
- **시스템으로서의 결과물** 강조 — 문제 인식 → 시스템 아키텍처 → 분석 프로세스 흐름
- **알고리즘 기반 분석 설명** — 오토인코더와 SHAP의 알고리즘 디테일
- **모델 성능 검증 + Validation 검증** 둘 다 — **운영 시나리오 정의** 후 입력 데이터 설명까지 포함
- **UI 다음 데모 영상** — 이상값 주입 시 감지 시연 (대시보드 기준)
- **백엔드 실시간/스트리밍 처리** 기술 설명 (통신 방식 포함)
- **MLOps 관점**의 자동화 프로세스 추가
- **기대효과 (정성/정량)** + **비즈니스 모델 활용 방안**

### 7-2. 반영 확정 사항
- **이상탐지는 AE 기반 다변량 탐지로 유지** ✅
- **알람 운영: (1-2-3) σ + 연속 N회 디바운싱**으로 설계 방향 확정 (현재 (2,3,6) 운영 상태에서 전환 예정)
- **운영 시나리오 정의 → F1 측정 시 시나리오 input data 설명** 포함

### 7-3. 남은 과제
- 디바운싱 N값 도메인별 튜닝 필요
- **NUTRIENT 도메인 오탐 근본 수정**이 알람 정확도 최대 변수

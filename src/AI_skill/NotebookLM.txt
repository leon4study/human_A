Session Close Ritual (self-contained). When the user signals session-end — "close out," "wrap up," "let's stop here," or /close — run a single terminal procedure that captures state using only the files and folders you create yourself. Do not assume any pre-existing infrastructure. If the user has no Continuity/ folder, no memory/ directory, and no prior reports, your job is to bootstrap the minimum on first use and keep it lightweight forever after. A close should always be possible from a cold start with nothing but Read and Write tools.

Step 1 — Check what exists, create only what's missing. Look in the current working directory for a file named SESSION_LOG.md. If it exists, you will append to it. If it does not, create it with a one-line header (# Session Log) and nothing else — no folders, no subdirectories, no ceremony. One file is the entire infrastructure. This keeps the ritual portable: it works in any project, on any machine, without setup.

Step 2 — Reflect. Scan the conversation for what worked, what failed, where the user corrected you, and where the user confirmed a non-obvious choice. Quiet confirmations are easy to miss and just as valuable as corrections. Do not skip this step because the session "felt routine"; routine sessions still produce calibration data.

Step 3 — Append a dated entry to SESSION_LOG.md. Add a new section with today's date and a session number (count existing entries + 1). Under it, record four things in plain prose: (a) what was accomplished this session, (b) what remains and the top 3–5 next tasks in priority order, (c) any durable lessons the user taught you — phrased as "always do X" or "never do Y" — with a one-line "why" for each, and (d) a one-line resume instruction the next session can act on. Treat the lessons subsection as mandatory: if you found none, write "No durable lessons this session" rather than omitting it. Append only; never rewrite or delete past entries, because the log's value is its history.

Step 4 — Close hard. End the conversation with a short summary showing the path to SESSION_LOG.md, the top remaining tasks, the resume instruction, and a clear "CLOSE PIPELINE COMPLETE" marker. After the marker, stop. No follow-up options, no "anything else?" The marker tells the next session not to re-run the close.

Why this matters. A self-sufficient close has no external dependencies to break: no plugin to install, no directory to pre-create, no schema to learn. One file, appended each session, readable by any future session (or any human) with nothing but a text editor. Start simple. If the log grows unwieldy, the user can split it later — but premature infrastructure is worse than a plain file that always works.
